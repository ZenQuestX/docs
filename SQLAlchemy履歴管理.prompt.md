# アーキテクチャと実装の制約条件

これからPostgreSQLとFastAPI(SQLAlchemy)を用いた履歴管理システムの実装を行います。
以下の設計思想と制約を絶対に厳守してコードを生成してください。

1. 【テーブル設計】Hub & Spokeパターン
- 属性を持たない「実体（Hub）テーブル」と、属性と期間を持つ「履歴（Spoke）テーブル」を分離すること。
- 履歴テーブルから実体テーブルへは明示的な外部キー（FK）を張り、`ondelete='RESTRICT'` を設定すること。
- 主キーはすべてUUIDを使用し、履歴テーブルの行が追加されるたびに新しいUUIDを発行すること。

2. 【データ整合性】PostgreSQLへの委譲
- アプリケーション側のバリデーションに依存せず、PostgreSQLの機能で矛盾を防ぐこと。
- 履歴テーブルには `tstzrange` 型の `valid_period` カラムを設けること。
- 同一実体に対する期間の重複を防ぐため、`EXCLUDE USING gist` による排他制約をDDL（及びSQLAlchemyの__table_args__）に必ず含めること。

3. 【更新ロジック】SCD Type 2の徹底
- 既存の履歴レコードに対する単純な属性のUPDATEは禁止。
- 状態を変更する際は、必ず同一トランザクション内で「現在の有効レコードの valid_period 終了日時を現在時刻でUPDATE（クローズ）」し、「新しいレコードを INSERT」すること。
- 物理削除（DELETE）は一切行わず、valid_period をクローズすることで論理削除（無効化）を表現すること。
