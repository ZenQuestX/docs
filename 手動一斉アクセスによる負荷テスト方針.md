### **【テンプレート】手動一斉アクセスによる負荷テスト方針**

**1. テストの目的**

本テストは、アプリケーション `[アプリケーション名]` のサービス公開に先立ち、多数のユーザーによる同時アクセスが発生した際のシステムの挙動およびユーザー体感を評価することを目的とする。
これにより、以下の点を確認する。

  * 高負荷時におけるサーバーダウンや致命的なエラーが発生しないこと（**可用性の確認**）
  * アクセス集中時におけるレスポンスの遅延が、ユーザー体験を著しく損なうレベルでないこと（**体感性能の確認**）
  * AWSのオートスケーリングなど、負荷に応じてインフラが正常にスケールするかどうかの挙動確認

`[補足：このテストはツールのよる精密な性能測定ではなく、あくまでユーザー視点でのサービスの安定性を確認する定性的な評価が主目的である]`

-----

**2. テストの対象範囲**

  * **対象システム**: `[アプリケーション名]`
  * **対象URL**:
      * `https://example.com/`
      * `https://example.com/login`
      * `https://example.com/products/123`
  * **対象外**:
      * 管理画面
      * 外部連携API `[例: 〇〇社の決済API]` への直接の負荷
      * バッチ処理などの非同期処理

-----

**3. テスト実施要項**

  * **実施日時**: `[YYYY年MM月DD日 HH:MM ～ HH:MM]`
  * **実施場所**: `[例: オンライン（各自のPCから）]`
  * **テスト環境URL**: `[テスト対象のURL]`
  * **参加人数**: `[XX 名]`

-----

**4. テスト手法**

  * 事前に定めたシナリオに基づき、指定時刻に参加者全員が一斉に手動でアクセスを行う（人海戦術）。
  * テストは複数回（例: 3回）に分けて実施し、徐々に負荷の高いシナリオへ移行する。

-----

**5. 役割分担**

| 役割 | 担当者 | 内容 |
| :--- | :--- | :--- |
| **指揮者** | `[氏名]` | 全体の号令、テスト開始・終了の合図 |
| **インフラ監視** | `[氏名]` | CloudWatch等でサーバーメトリクスをリアルタイム監視 |
| **テスト参加者** | 全員 | 事前に割り当てられたシナリオを実施し、結果を報告 |
| **結果取りまとめ** | `[氏名]` | 各参加者からのフィードバックを集約し、レポートを作成 |

-----

**6. テストシナリオ**

参加者は以下のシナリオを、指揮者の合図で実行する。

  * **第1回 (10:00～)**: **基本閲覧シナリオ**
    1.  トップページ `[URL]` にアクセスする。
    2.  商品一覧ページ `[URL]` にアクセスする。
    3.  任意の商品を1つ選び、詳細ページにアクセスする。
  * **第2回 (10:15～)**: **検索シナリオ**
    1.  トップページから `[検索キーワード]` で商品を検索する。
    2.  検索結果から任意の商品詳細ページにアクセスする。
  * **第3回 (10:30～)**: **[より負荷のかかるシナリオ、例: ログイン処理など]\`**
    1.  ログインページ `[URL]` にアクセスし、指定のアカウントでログインする。
    2.  マイページ `[URL]` を表示する。

-----

**7. 評価基準（ゴール）**

以下の状態をもって、本テストは合格とする。

  * **システム**:
      * テスト中に5xx系のサーバーエラーが多発しないこと。（許容エラー率: `[例: 1%未満]`）
      * サーバーがダウンし、アクセス不能状態にならないこと。
  * **ユーザー体感**:
      * 参加者の半数以上が「ページの表示が極端に遅い（`[例: 10秒以上]`）」と感じないこと。
  * **インフラ**:
      * CloudWatch上で、CPU使用率やメモリ使用率が危険水域（`[例: 90%]`）に張り付かないこと。
      * 負荷に応じて、Fargateタスク数やLambda同時実行数がスケールすることが確認できること。

-----

**8. 監視項目**

インフラ監視担当は、テスト中に以下のCloudWatchメトリクスを注視する。

  * **主要メトリクス**:
      * `[例: ALBのHTTPCode_Target_5XX_Count]`
      * `[例: ECS/FargateのCPUUtilization, MemoryUtilization]`
      * `[例: RDSのCPUUtilization, DBConnections]`
      * `[例: LambdaのErrors, Throttles]`

-----

**9. 結果報告方法**

  * **参加者**:
      * テスト中に発生したエラーや、著しい表示遅延を感じた場合は、スクリーンショットを撮影する。
      * 各回のテスト終了後、指定のチャットツール `[例: Slackの #testチャンネル]` にて、以下の形式で体感を報告する。
        > ```
        > 第〇回：[快適 / 少し遅い / エラー]
        > 気づいた点：(あれば記入)
        > ```
  * **結果取りまとめ担当**:
      * テスト終了後、`[YYYY年MM月DD日]` までに、参加者のフィードバックとCloudWatchのメトリクスを突き合わせた結果レポートを作成する。

-----

このテンプレートをベースに、ご自身のプロジェクトの状況に合わせて自由にカスタマイズしてください。ドキュメントとして方針を明確にすることで、参加者全員の目線が合い、スムーズなテスト実施につながります。
