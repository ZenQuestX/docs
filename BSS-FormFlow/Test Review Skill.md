Test Review Skill
このスキルは、実装コードとテストコードの組を評価し、次のフェーズに進んでよいかを判定するための基準を定義する。 呼び出し側(ワークフロー)は本スキルにレビュー実行を委譲し、末尾の GATE_RESULT のみを見て次のアクションを決めてよい。
--------------------------------------------------------------------------------
入力
呼び出し側は以下を提供すること。

    完了条件(Definition of Done): 当該フェーズで満たすべき条件の箇条書き
    実装コード: 対象ファイルの差分または全文
    テストコード: 対象ファイルの差分または全文
    (任意)直前フェーズまでの既存テスト一覧: リグレッション確認用

いずれかが欠落している場合、レビューは実行せず GATE_RESULT: FAIL とし、FAIL_REASONS に「入力不足: 何が足りないか」を記載する。
--------------------------------------------------------------------------------
評価軸
以下6軸それぞれについて、「該当なし」または「要修正」を判定する。要修正の場合は該当箇所(ファイル名・行・コード引用)と理由を必ず添える。
軸1: 検出力(Fault Detection)
問い: このテストは、対応する実装が壊れたとき本当にRedになるか。 チェックするパターン:

    例外を握りつぶし、失敗として扱っていない(try/catch で何もしない、except: pass 等)
    not null / toBeDefined() など、値の中身を見ない緩いassertion
    モックの戻り値をそのままassertしているだけで、実装ロジックを経由していない

軸2: リスク対応度(Coverage vs Impact)
問い: 完了条件に関わる分岐(異常系・境界値・リトライ・タイムアウト等)がテストされているか。 チェックするパターン:

    実装コード中の if / catch / リトライ / タイムアウト分岐を列挙し、対応するテストケースが存在するか突き合わせる
    カバーされていない分岐があれば具体的に列挙する

軸3: 独立性・再現性(Isolation & Determinism)
問い: 実行順序、グローバル状態、時刻、乱数、非同期タイミングに依存していないか。 チェックするパターン:

    固定 sleep / setTimeout による同期待ち
    共有状態(モジュール変数・DB等)を初期化せず前提にしている
    Date.now() / Math.random() を直接使用し、モック・固定化されていない

軸4: 粒度・意図の明確さ(Single Responsibility & Readability)
問い: 1テストが1つの関心事を検証しているか。失敗時に原因が一目でわかるか。 チェックするパターン:

    1テスト内で異なる関心事のassertionが混在していないか
    テスト名から「何が壊れるとFailするか」が読み取れるか

軸5: 脆弱性(Fragility to Refactoring)
問い: 外部から見た振る舞いが変わらないリファクタリングでも壊れる作りになっていないか。 チェックするパターン:

    モック検証が「呼ばれたか」ではなく「回数・順序」まで過剰に固定していないか
    プライベートな実装詳細に依存したassertionがないか

軸6: コスト妥当性(Cost/Benefit)
問い: テストの実行コスト・複雑さは、守っている価値に見合っているか。 チェックするパターン:

    セットアップ(モック定義等)が過大な割に、検証しているロジックが些細でないか

--------------------------------------------------------------------------------
判定ポリシー

    軸1・軸2で「要修正」が1件でもある場合 → GATE_RESULT: FAIL(検出力とリスクカバレッジは妥協しない)
    軸3・軸5で「要修正」がある場合 → 原則 FAIL。ただし影響範囲が軽微(既存テストへの影響なし、flaky実績なし)と判断できる場合のみ PASS とし、NOTES に理由を明記してよい
    軸4・軸6は「要修正」があっても即FAILにはしない。NOTES に指摘を残し PASS としてよい(可読性・コストは次フェーズ以降で改善可能なため)
    完了条件(Definition of Done)を満たしていない項目が1つでもあれば、他の軸の結果によらず GATE_RESULT: FAIL
    判断に迷うケース(情報不足で確証が持てない等)は保守的に FAIL 側に倒す

--------------------------------------------------------------------------------
出力形式
以下のフォーマットで出力すること。

## 完了条件の充足
- [条件1]: 満たしている / 満たしていない(理由)
- [条件2]: ...

## 軸1: 検出力
該当なし / 要修正
(要修正の場合、ファイル名・行・コード引用・理由)

## 軸2: リスク対応度
...

## 軸3: 独立性・再現性
...

## 軸4: 粒度・意図の明確さ
...

## 軸5: 脆弱性
...

## 軸6: コスト妥当性
...

## NOTES
(軸4・軸6の軽微な指摘、または軸3・軸5でPASSとした場合の理由など)

GATE_RESULT: PASS または FAIL
FAIL_REASONS:
- (FAILの場合のみ、修正すべき項目を箇条書き。PASSの場合はこのセクション自体を出力しない)

GATE_RESULT と FAIL_REASONS の行は、呼び出し側(ワークフロー)が機械的にパースする前提のため、文言・書式を変更しないこと。
--------------------------------------------------------------------------------
禁止事項

    完了条件の充足を確認せずに軸1〜6だけで PASS を出さないこと
    実装者(自分自身)が書いたコードに対して、正当化のために基準を緩めないこと。曖昧な場合はFAIL側に倒す
    該当なしの軸について、無理に指摘を作らないこと(該当なしはそのまま「該当なし」と書く)
