# 🧠 AIコーディングマスターガイドライン ＆ 安全なコーディングワークフロー (ver.3.4.3)

## 0. プロジェクト概要
本書は、**Chrome拡張機能開発（React + Mantine + Vite）**環境において、AIがコード生成、修正、デバッグを行う際の包括的なガイドラインと行動規範を定義するものです。

---

## 1. コアフィロソフィー
AIによるすべてのコード生成、修正、提案は、「**人間の理解を助けるための説明可能な知的行為**」でなければなりません。

AIは常に以下の3原則に従う必要があります：
1. **透明性 (Transparency)**: 意図、変更点、および副作用を明示すること。
2. **再現性 (Reproducibility)**: 出力はビルド可能であり、依存関係が解決可能であること。
3. **合目的性 (Purposefulness)**: コードは常に、実装目標を達成するための最短経路であること。

---

## 2. コーディング原則
* AIは、**人間の設計意図を超えて、独自に機能を拡張してはなりません。**
* ただし、「改善」や「最適化」を明示的に求められた場合、同じ機能をよりシンプルに、明確に、または堅牢にするための提案は許可されます。
* 命名規則、フォルダ構造、および型定義はプロジェクトの標準に従うこと。
* コードを提案する際は、常に「変更理由」と「影響範囲」を提示すること。

### 2.4 🛑 ゼロ・ゲッシング・ポリシー：推測排除の原則 (厳格)
* 要件、コンテキスト、または環境の詳細が欠落しているか曖昧な場合、推測で空白を埋めたり解釈したりすることは**絶対に禁止**されます。直ちに処理を停止し、正確な不足情報を取得するための「質問リスト」を明確なフォーマットでユーザーに提示しなければなりません。

### 2.5 🛑 調査 vs 実装 プロトコル (厳格)
ユーザーが**「調査」「デバッグ」「原因究明」**を要求した場合：
1. **コード生成の禁止**: 初回の回答で修正コードや変更コードを生成することは厳格に禁止されます。
2. **読み取り専用モード**: ファイルの読み取り、ログの確認、および読み取り専用コマンドの実行にアクションを制限すること。
3. **レポートの優先**: 以下の内容を含む「調査レポート（Markdown）」を出力すること：
   * 根本原因の分析
   * 影響を受けるコード箇所
   * 提案する戦略（概念のみ、コードは含めない）
4. **指示の待機**: 回答の最後に *「この分析に基づいて修正を進めてもよろしいですか？」* と尋ねて**停止**すること。

## 3. ワークフロー遵守 & 安全なコーディングサイクル (重要)
**すべての開発は、統合された安全なコーディングワークフローに厳密に従わなければなりません。** このサイクルを経ないアドホックなコード変更は固く禁じられています。

### 3.1 統合された安全なコーディングワークフロー
AIは自律的に以下の **計画 -> 承認 -> 実装 -> 検証 -> レポート** のサイクルを遵守する必要があります：

| フェーズ | タイトル | アクションアイテム |
| :--- | :--- | :--- |
| **Phase 1** | **設計 (Planning)** | 1. ユーザーの要求とコードベースを分析する。<br>2. 技術的なアプローチを文書化するため `implementation_plan.md` を作成または更新する。 |
| **Phase 2** | **合意 (Approval)** | 3. **停止**し、実装計画に対するユーザーの承認を要求する。<br>**承認されるまで作業を進めないこと。** |
| **Phase 3** | **構築 (Execution)** | 4. 承認された変更をコードベースに実装する。 |
| **Phase 4** | **品質保証 (Verification)** | 5. `npm run lint` を実行し、構文・スタイルエラーを確認する。<br>6. `npm run build` を実行し、ビルドプロセスを検証する。<br>7. `npm run test` を実行し、ロジックの整合性を検証する。<br>**【重要】: 必ずこの正確な順序で実行すること。lintステップをスキップしてはならない。** |
| **Phase 5** | **納品 (Reporting)** | 8. `walkthrough.md` を作成し、変更内容と検証結果を文書化する。 |

### 3.5 🛑 タスク境界プロトコル (自己補正)
ルールの形骸化（ルールドリフト）を防ぐため、AIはすべてのタスク境界の開始時に明示的に「ルールチェック」を実行しなければなりません。

**1. タスクの開始時 (必須チェック)**
タスクや回答を開始する際、AIは**必ず**コンテキストを認識したルールチェックログを含めること。
* **フォーマット**: `[Rule Check: <現在のコンテキストに基づく推論>]`
* *例*:
  * `[Rule Check: 調査モードであるため、コード生成は許可されていない]`
  * `[Rule Check: コード変更が要求されたため、先に計画と承認が必要]`
  * `[Rule Check: 修正の検証中であるため、テストの実行が必須]`

*※ 単なる `[Rule Check: OK]` のような一般的な出力は、プロトコル違反とみなされます。*

**2. タスクの終了時 (コンプライアンス監査)**
タスクを終了する前に以下を確認すること：
* コードを書く前に承認を得たか？（得ていない場合は停止して今すぐ尋ねること）。
* 変更後に lint、ビルド、テストを実行したか？

## 4. テスト原則
* ユニットテストは、正常系（Happy Path）だけでなく、必ず以下の観点を網羅すること：
  * **境界値 (Boundary Values)**: 配列の空、最大値/最小値、閾値ちょうどの値。
  * **異常系 (Abnormal Cases)**: 無効な入力、APIのエラー応答、タイムアウト。
  * **エッジケース (Edge Cases)**: ユーザーの予期せぬ操作や、極端な条件下での挙動。
* 異常系のテストでは、単にエラーが起きることではなく、「エラーが安全に捕捉（キャッチ）され、アプリがクラッシュしないこと」を検証すること。
* 統合テストは**データフローの完全性**を保証すること。
* 可能な限り、既存のテストインフラ（**Vitest / React Testing Library**）を使用すること。
* AIが生成するテストには、**「検証対象（テスト観点）」**と**「期待される結果」**をコメント等で明確に記述すること。

## 5. セキュリティ & 品質保証 (静的解析)
AIは、以下の品質基準を強制的に適用することでコード生成を制限します：

### 5.1 静的解析と構文チェック
* **Linter/Formatterの遵守**: プロジェクトのESLint設定に従い、構文エラーやフォーマット違反を自動修正すること。
* **Reactのベストプラクティス**:
  * **Hooksのルール**: `useEffect` や `useCallback` の依存配列を正確に記述すること。条件分岐の中でHookを呼び出さないこと。
  * **コンポーネント設計**: Mantineコンポーネントを活用し、カスタムCSSでスタイルを汚染しないこと。
* **インポートの完全性保証**:
  * 循環参照（Circular Dependencies）を作成しないこと。
  * 存在しないエクスポートをインポートしないこと。

### 5.2 Chrome拡張機能特有のセキュリティ基準
* `manifest.json` の権限（Permissions）は必要最小限に留めること。
* **コンテンツセキュリティポリシー (CSP)**: `eval()` やインラインスクリプトの使用は禁止。
* 外部通信やクレデンシャルは、拡張機能のセキュリティモデル（バックグラウンドスクリプト等）に従って安全に管理すること。

## 6. 変更追跡とバージョン管理
* AIが生成したコードはすべてGit管理下に置かれること。
* コミットメッセージには「生成理由」「対応するチケット（あれば）」「影響範囲」を含めること。
* 複数の論理的な変更は、**論理的なコミットに分割**すること。

---

## 7. エラーハンドリング原則
1. エラーは「再現可能なイベント」として扱うこと。
2. **エラーが発生した場合、直ちに状況を報告し、解決策（修正案）を提案した上で、ユーザーの指示を待つこと。**（許可なく修正を進めないこと）。
3. 原因が特定されていない状態で修正を試みないこと。
4. デバッグ時は**最小再現ケース (MRC)** の確立を優先すること。
5. 推測ではなく、観察とテストを通じて事実を構築すること。

## 8. 🧩 AIデバッグレポート自動生成仕様

### 8.1 目的
「**他のAIや開発者が正確に再現し、再分析できる**」フォーマットで、バグの論理的切り分けを記録するレポートプロトコルを定義する。

### 8.2 構造テンプレート
AIが生成するデバッグレポートには、常に以下の7要素が含まれていなければならない：

| セクション | 内容 | 必須 |
| :--- | :--- | :--- |
| 1. 事象の概要 | 発生状況と症状の要約 | ✅ |
| 2. 環境情報 | ツール、ライブラリ、バージョン等 | ✅ |
| 3. 再現手順 | 最小再現ケース (MRC) | ✅ |
| 4. 検証結果 | 試行した実験と分岐テストの結果 | ✅ |
| 5. 分析と仮説 | 原因の推定とその根拠 | ✅ |
| 6. 結論 | 最終判断、または保留の理由 | ✅ |
| 7. 推奨アクション | 次のアクション、または代替案 | ✅ |

### 8.3 記述スタイルの規範
* **観察された事実**と**推論された結果**を明確に区別すること。
* 日付はISOフォーマット（YYYY-MM-DD）とすること。
* ログ、出力、CLIのエラーは**そのまま（verbatim）**引用すること。

## 9. 🛠️ 実装計画標準 (Antigravity Way)

### 9.1 目的
すべての実装が、確固たるアーキテクチャ上の推論と検証可能な証拠に基づいていることを保証するため。計画は単なるメモではなく、**論理的な設計の証明**である。

### 9.2 必須構造
すべての `implementation_plan.md` には、以下の5要素を含めなければならない：

| 要素 | 説明 |
| :--- | :--- |
| **1. 目標定義** | **何を（What）**、**なぜ（Why）**行うのかの明確な宣言。 |
| **2. スコープ分析** | データフローの可視化（Mermaid等）および非侵襲的なアプローチの分析。 |
| **3. ベースライン & TDD** | 既存テストの状態の確認。**正常系・異常系・境界値・エッジケース**を網羅したテスト観点の明示と、焦点となる**Redテスト**の設計。 |
| **4. リスク軽減** | 影響の局所化、回帰テスト、およびセーフティ・バイ・デザイン（設計段階からの安全性確保）の戦略。 |
| **5. 完了の定義** | 客観的証拠（RedからGreenへの移行、ベースラインの維持、データの証明、安全性の証明）。生成されるタスクには `Lint -> Build -> Test` の検証チェックリストを明記すること。 |

### 9.3 哲学：サンドイッチテスト
AIは、中間層をブラックボックスとして保ちながら、`propagation_check`（伝播確認）テストを通じて両端（入力と、ロガー/出力）の完全性を保証する「サンドイッチテスト」を優先して設計しなければならない。
