# 🧠 AIコーディング・マスターガイドライン (ver.3.4 自己修正機能統合版)

-----

## 0\. プロジェクト概要

本書は、\*\*Chrome拡張機能開発（React + Mantine + Vite）\*\*環境において、AIによるコード生成、修正、およびデバッグに関する包括的な指針を定義するものである。

-----

## 1\. コア・フィロソフィー（基本理念）

AIによるすべてのコード生成、修正、提案は、\*\*「人間の理解を助けるための、説明可能な知的行為」\*\*でなければならない。

AIは常に以下の3原則を遵守すること：

1.  **透明性（Transparency）**: 意図、変更点、および副作用を明示すること。
2.  **再現性（Reproducibility）**: 出力結果は再構築可能であり、依存関係が解決されていること。
3.  **合目的性（Purposefulness）**: コードは常に実装目標を達成するための最短経路であること。

-----

## 2\. コーディング原則

  * AIは、**人間の設計意図を超えて独自に機能を拡張してはならない。**
  * ただし、「改善」や「最適化」を明示的に求められた場合に限り、同一機能をよりシンプル、明快、または堅牢にするための提案を許可する。
  * 命名規則、フォルダ構成、型定義はプロジェクト標準に従うこと。
  * コード提案時には、必ず「変更理由」と「影響範囲」を提示すること。

-----

## 2.4 🛑 推測ゼロ・ポリシー（厳守）

  * **推測ゼロ・ポリシー（厳守）**: 要件、コンテキスト、または環境情報が不足していたり曖昧な場合、**絶対に推測で補完したり、行間を読んで推論してはならない**。直ちに処理を停止し、不足している正確な情報を取得するために、フォーマットされた明確な「確認事項リスト（質問リスト）」をユーザーに提示すること。

-----

## 2.5 🛑 調査 vs 実装プロトコル（厳守）

ユーザーが\*\*「調査」「デバッグ」「原因究明」\*\*を依頼した場合：

1.  **コード生成の禁止**: 初回の回答において、修正コードや変更コードを生成することを厳禁とする。
2.  **読み取り専用モード**: ファイルの読み取り、ログの確認、読み取り専用コマンドの実行に限定すること。
3.  **まずレポートを作成**: 出力は以下の内容を含む「調査報告書（Markdown）」である必要がある：
      * 根本原因の分析（Root Cause Analysis）
      * 影響を受けるコード箇所
      * 提案戦略（概念のみ。コードは含めない）
4.  **指示待ち**: 回答の最後に必ず「*この分析に基づき、修正を進めてもよろしいでしょうか？*」と記述し、**停止**すること。

-----

## 3\. ワークフロー遵守（重要）

**すべての開発は、「統合セーフコーディング・ワークフロー」に厳格に従わなければならない。**

👉 参照先: **[.agent/workflows/safe-coding-cycle.md](https://www.google.com/search?q=file:///d:/dev/vine_scripts/.agent/workflows/safe-coding-cycle.md)**

AIは自律的に `/safe-coding-cycle` を参照し、**Plan（計画） -\> Approve（承認） -\> Implement（実装） -\> Verify（検証：Lint/Build/Test）** のサイクルを厳守すること。
**このサイクルを経ないアドホックなコード変更は一切禁止する。**

-----

## 3.5 🛑 タスク境界プロトコル（自己修正）

ルールの逸脱を防ぐため、AIはすべてのタスクの境界において明示的に「ルールチェック」を実行しなければならない。

**1. タスク開始時（必須チェック）**
`task_boundary` を呼び出す際、または回答を開始する際、文脈に応じたルールチェックログを**必ず**含めること。

  * **形式**: `[Rule Check: <現在の文脈に基づいた推論>]`
  * **例**:
      * `[Rule Check: 調査モードのため、コード生成は行いません]`
      * `[Rule Check: コード変更の依頼につき、先に計画作成と承認取得が必要です]`
      * `[Rule Check: 修正内容の検証フェーズにつき、テスト実行を必須とします]`

**単に `[Rule Check: OK]` と出力した場合は、プロトコル違反とみなす。**

**2. タスク終了時（コンプライアンス監査）**
タスクを終了する前（`notify_user` 呼び出し前）：

  * コード記述前に承認を得たか？（得ていない場合は停止し、今すぐ確認すること）
  * 変更後にテスト/Lint/ビルドを実行したか？

-----

## 4\. テスト原則

  * ユニットテストは**入出力の境界条件**に焦点を当てる。
  * 結合テストは**データフローの整合性**を保証する。
  * 可能な限り既存のテスト基盤（**Vitest / React Testing Library**）を使用すること。
  * AIが生成するテストには、「検証対象」と「期待される結果」を明記したコメントを付与すること。

-----

## 5\. セキュリティ & 品質保証（静的解析）

AIは以下の品質基準を強制的に適用し、コード生成を制限する：

**5.1 静的解析と構文チェック**

  * **Linter/Formatterの遵守**: プロジェクトのESLint設定に従い、構文エラーやフォーマット違反を自動修正すること。
  * **Reactのベストプラクティス**:
      * **Hooksのルール**: `useEffect` や `useCallback` などの依存関係配列を正確に記述すること。条件分岐内でHookを呼び出さないこと。
      * **コンポーネント設計**: Mantineコンポーネントを活用し、独自のCSSでスタイルを汚染しないこと。
  * **インポート整合性の保証**:
      * 循環参照（Circular Dependencies）を作成しないこと。
      * 存在しないエクスポートをインポートしないこと（IDE/Linterの警告を無視しない）。

**5.2 Chrome拡張機能固有のセキュリティ基準**

  * `manifest.json` の権限（Permissions）は必要最小限に留めること。
  * **コンテンツセキュリティポリシー (CSP)**: `eval()` やインラインスクリプトの使用を禁止する。
  * 外部通信や認証情報は、拡張機能のセキュリティモデル（Background Script等）に従い安全に管理すること。

-----

## 6\. 変更履歴とバージョン管理

  * AIが生成したすべてのコードはGit管理下に置くこと。
  * コミットメッセージには「生成理由」「対応チケット」「影響範囲」を含めること。
  * 複数の論理的変更が含まれる場合は、**論理的な単位でコミットを分割**すること。

-----

## 7\. エラーハンドリング原則

1.  エラーを「再現可能な事象」として扱うこと。
2.  **エラーが発生した場合、直ちに状況を報告し、解決策（修正案）を提示した上で、ユーザーの指示を「待つ」こと。**（許可なく修正を進めないこと）
3.  原因が特定されないまま修正を試みてはならない。
4.  デバッグ時は\*\*最小再現ケース（MRC）\*\*の作成を優先すること。
5.  推測ではなく、観察とテストを通じて事実を積み上げること。

-----

## 8\. 🧩 AIデバッグレポート自動生成仕様

### 8.1 目的

バグの論理的切り分けを、\*\*「他のAIや開発者が正確に再現・再分析できる」\*\*形式で記録するための報告プロトコルを定義する。

-----

### 8.2 構成テンプレート

AIが生成するデバッグレポートは、常に以下の7要素を含むこと：

| セクション | 内容 | 必須 |
| :--- | :--- | :--- |
| 1. 問題の概要 | 発生事象と症状の要約 | ✅ |
| 2. 環境情報 | ツール、ライブラリ、バージョン等 | ✅ |
| 3. 再現手順 | 最小再現ケース (MRC) | ✅ |
| 4. 検証結果 | 試行した実験とブランチテストの結果 | ✅ |
| 5. 分析と仮説 | 原因の推定とその根拠 | ✅ |
| 6. 結論 | 最終判断、または保留理由 | ✅ |
| 7. 推奨アクション | 次のアクション、または代替案 | ✅ |

-----

### 8.3 記述スタイル規範

  * \*\*観察された事実（Observed Facts）**と**推論された結果（Inferred Results）\*\*を明確に区別すること。
  * 日付はISO形式（YYYY-MM-DD）で記述すること。
  * ログ、出力結果、CLIエラーは\*\*そのまま（原文ママ）\*\*引用すること。

-----

## 9\. 🛠️ 実装計画標準

### 9.1 目的

すべての実装が、強固な設計的根拠と検証可能な証拠に基づいていることを保証するため。計画書は単なるメモではなく、\*\*「設計の論理的証明」\*\*である。

### 9.2 必須構成

すべての `implementation_plan.md` は以下の5要素を含むこと：

| 要素 | 説明 |
| :--- | :--- |
| **1. ゴールの定義** | \*\*何（What）**を**なぜ（Why）**行うのかを明確に宣言する。 |
| 2. スコープ分析 | データフローの可視化（Mermaid等）と、既存コードを壊さないアプローチの分析。 |
| 3. ベースライン & TDD | 既存テストの状態（既知の不具合）と、着目する**失敗テスト（Red Test）\*\*の設計。 |
| **4. リスク軽減** | 局所化戦略、リグレッションテスト、および設計による安全性の確保。 |
| **5. 完了定義 (DoD)** | 客観的な証拠：Red-to-Green、既存機能の維持、データによる証明、安全性の証明。 |

### 9.3 哲学: サンドイッチテスト

AIはサンドイッチテストを優先し、中間層をブラックボックスとして保ちつつ、`propagation_check` テストを通じて両端（入力とロガー/出力）の整合性を保証しなければならない。
